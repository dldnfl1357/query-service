# 02. 어떻게 동작하나 — 큰 그림

## 핵심 부품 3가지

쿼리 서비스는 크게 **3가지 부품** 으로 이루어져 있습니다.

```
   [도메인 서비스들]          [우체통]            [쿼리 서비스]
   ┌──────────┐              ┌──────┐         ┌──────────────┐
   │ 회원 서비스 │ ──메시지──→ │ SQS  │ ──→ │ 1) 받기 (핸들러) │
   ├──────────┤              ├──────┤         ├──────────────┤
   │ 상품 서비스 │ ──메시지──→ │ SQS  │ ──→ │ 2) 적기 (DB write) │
   ├──────────┤              ├──────┤         ├──────────────┤
   │ 주문 서비스 │ ──메시지──→ │ SQS  │ ──→ │ 3) 보여주기 (API)  │
   ├──────────┤              ├──────┤         └──────┬───────┘
   │ 배송 서비스 │ ──메시지──→ │ SQS  │                │
   └──────────┘              └──────┘                ↓
                                                   [화면]
```

### 부품 1: SQS — 우체통

**SQS (Simple Queue Service)** 는 AWS가 제공하는 메시지 우체통입니다.

- 보내는 사람(예: 회원 서비스)이 편지(메시지)를 우체통에 넣어요.
- 받는 사람(쿼리 서비스)이 우체통을 열어서 편지를 한 통씩 꺼냅니다.
- 받는 사람이 잠시 자리를 비워도 편지는 우체통에 안전하게 보관됩니다.

**왜 직접 호출하지 않고 우체통을 거치나?**

- 회원 서비스가 멈춰도 우체통은 남아 있으니 메시지가 사라지지 않음
- 쿼리 서비스가 잠깐 느려도 우체통에 쌓아두면 되니까 회원 서비스가 막히지 않음
- 두 서비스가 **느슨하게 연결** 되어서 한쪽 문제가 다른 쪽으로 안 번짐

**우리는 도메인마다 우체통을 따로 둡니다.**

- `order-changed` 우체통 — 주문 변경 알림 전용
- `member-changed` 우체통 — 회원 변경 알림 전용
- `product-changed` 우체통 — 상품 변경 알림 전용
- `delivery-changed` 우체통 — 배송 변경 알림 전용

### 부품 2: 비정규화 테이블 — 미리 정리된 영수증

데이터베이스(MySQL)에 **`order_list_view`** 라는 테이블이 하나 있습니다.
이름이 좀 어렵지만, 그냥 **"주문 목록 화면 한 줄에 들어갈 모든 정보를 한 줄에 다 넣은 테이블"** 입니다.

```
order_list_view 테이블 (한 줄 = 한 주문)
┌──────────┬──────────┬───────────┬──────────┬──────────┐
│ order_id │ 회원이름   │ 상품이름    │ 가격      │ 배송상태   │
├──────────┼──────────┼───────────┼──────────┼──────────┤
│  1001    │ 홍길동    │ 갤럭시폰    │ 1,200,000│ 배송중    │
│  1002    │ 김철수    │ 라면 한박스 │   15,000│ 배송완료   │
└──────────┴──────────┴───────────┴──────────┴──────────┘
```

원래 같으면 회원, 상품, 주문, 배송 4개의 테이블에 흩어져 있을 정보입니다.
그걸 **하나의 테이블에 베껴서 정리해 둔** 것이라 "비정규화(denormalized)" 라고 부릅니다.

### 부품 3: 화면용 API — 보여주기 창구

화면(웹/앱)이 `/api/order-list` 같은 주소로 물어보면 이 테이블만 읽어서 돌려줍니다.
다른 서비스로 한 번도 안 나갑니다. 그래서 빠릅니다.

## 데이터가 흘러가는 과정

회원 정보 한 건이 바뀐다고 가정해 봅시다.

1. 회원 서비스에서 홍길동의 등급이 "실버 → 골드" 로 변경됨.
2. 회원 서비스가 `member-changed` 우체통에 "홍길동(id=42) 등급=골드" 라는 편지를 넣음.
3. 쿼리 서비스가 우체통을 보고 편지를 꺼냄.
4. 쿼리 서비스는 자기 테이블에서 `회원ID=42` 인 모든 주문 줄을 찾아서 회원 등급을 골드로 업데이트.
5. 끝. 이제 화면에서 홍길동 주문을 보면 골드로 보임.

```
회원 서비스: "홍길동 골드 됐어!"
                 ↓ (우체통)
쿼리 서비스: "어디 보자... 홍길동(id=42) 주문이 3건 있네, 다 골드로 바꿈"
```

## 1 화면 = 1 테이블 원칙

**이게 가장 중요한 규칙입니다.**

화면이 늘어나면 테이블도 늘어납니다.

- 주문 목록 화면 → `order_list_view` 테이블
- 상품 상세 화면 → `product_detail_view` 테이블
- 회원 마이페이지 → `member_my_page_view` 테이블
- ...

같은 회원 정보가 여러 테이블에 베껴져 있어도 괜찮습니다.
오히려 **각 화면이 자기 모양에 최적화된 테이블을 갖는 것** 이 빠른 조회의 비결입니다.

## 가상 스레드 — 일꾼을 많이 쓰는 마법

자바 21부터 **가상 스레드(Virtual Thread)** 라는 기능이 생겼습니다.
일꾼(스레드)을 거의 무한대로 만들 수 있어서, 여러 메시지를 동시에 처리하기 좋습니다.
`application.yml` 의 `spring.threads.virtual.enabled: true` 한 줄로 켭니다.

쉽게 말해: 우체통에서 편지 100통이 한꺼번에 와도 100명의 일꾼이 동시에 처리.

## 한 줄 요약

- 도메인 서비스가 우체통(SQS)에 변경 알림을 넣고
- 쿼리 서비스가 그걸 받아서 미리 정리된 테이블(비정규화)에 적어두고
- 화면은 그 테이블만 읽어서 빠르게 보여줌
