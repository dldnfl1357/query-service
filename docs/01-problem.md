# 01. 왜 만들었나 — 문제 상황

## 마트 영수증 비유

큰 마트에 가서 장을 봤다고 생각해 봅시다. 마트에는 코너가 여러 개입니다.

- 과일 코너
- 정육 코너
- 생선 코너
- 음료 코너

손님이 계산대에서 "내가 산 모든 물건이 정리된 영수증 화면을 보여주세요" 하고 말했습니다.

만약 영수증을 만들 때마다 직원이 **각 코너를 매번 뛰어다니면서** 가격을 물어본다면?

- 과일 코너로 뛰어감 → 사과 가격 받음
- 정육 코너로 뛰어감 → 소고기 가격 받음
- 생선 코너로 뛰어감 → 고등어 가격 받음
- 음료 코너로 뛰어감 → 콜라 가격 받음
- 다시 자리로 돌아와서 영수증 작성

매번 이렇게 하면 손님은 영수증을 받기 위해 한참 기다려야 합니다. 손님이 많으면 마비 됩니다.

## 우리 회사도 똑같은 일이 일어났습니다

요즘 서버는 보통 **MSA(Micro Service Architecture, 마이크로서비스 구조)** 로 만듭니다.
"한 덩어리짜리 큰 프로그램" 대신 **부서별로 작은 프로그램들을 따로따로** 운영하는 방식입니다.

우리 회사를 예로 들면:

- **회원 서비스** — 회원 이름, 등급 같은 정보 관리
- **상품 서비스** — 상품 이름, 가격 정보 관리
- **주문 서비스** — 누가 무엇을 주문했는지 관리
- **배송 서비스** — 배송 상태 관리

이제 화면에서 "주문 목록"을 보여주려고 합니다. 한 줄에 다음이 다 있어야 합니다.

```
[주문번호] [회원이름] [상품이름] [가격] [배송상태]
```

이걸 만들려면 4개 서비스를 다 호출해서 데이터를 받아와야 합니다.
**한 명의 주문 = 4번의 통신.** 화면에 100건이 보인다면? 1000건이라면?
주문 한 건당 4번씩 4000번 통신 필요. 너무 느립니다.

게다가 각 서비스에는 수십만 ~ 수백만 건의 데이터가 들어 있습니다.
그걸 다 끌어와서 한 화면용으로 짜맞추는 일을 매번 한다는 건 비효율입니다.

## 그래서 만든 것이 "쿼리 서비스"

**아이디어:** 각 코너(서비스)에 매번 물어보지 말고, **미리 영수증 한 장을 만들어 두자.**

물건이 입고/가격변경 될 때마다 그 코너가 우리한테 "이거 바뀌었어!" 라고 알려줍니다.
우리는 그 정보를 받아서 **영수증 모양 그대로 정리된 테이블** 에 미리 적어 둡니다.
화면이 필요할 때는 그 테이블 한 곳만 보면 됩니다.

```
[기존 방식]
화면 ─→ 회원 서비스
   └─→ 상품 서비스    (느림, 매번 4번)
   └─→ 주문 서비스
   └─→ 배송 서비스

[새 방식]
회원 서비스 ─┐
상품 서비스 ─┤
주문 서비스 ─┤─→ 쿼리 서비스 (미리 정리해둔 테이블) ─→ 화면 (빠름, 한 번)
배송 서비스 ─┘
```

## 한 줄 요약

여러 부서의 데이터를 화면 모양 그대로 미리 정리해두는 "조회 전용 창고"가 쿼리 서비스입니다.
