# 04. 안전장치 — DLQ와 멱등성

메시지가 오가다 보면 별의별 일이 다 일어납니다.

- 편지 내용이 망가져서 도저히 읽을 수가 없을 때
- 같은 편지가 두 번 도착했을 때
- 편지가 거꾸로 도착했을 때 (늦게 보낸 게 먼저 옴)

이 셋을 어떻게 막는지 차례로 봅니다.

---

## 1. 망가진 편지 — DLQ (Dead Letter Queue)

### 문제

회원 서비스가 실수로 이상한 JSON 을 보냈습니다. 쿼리 서비스가 받아서 처리하려는데 에러가 납니다.
SQS 는 "처리 실패했으면 잠시 후 다시 줄게" 라고 생각하고 다시 줍니다.
다시 줘도 같은 에러. 또 다시 줘도 같은 에러. **무한 반복.**

이런 편지를 **독약 메시지(poison pill)** 라고 부릅니다.

### 해결: 5번 실패하면 따로 보관

AWS 의 SQS 에는 **재시도 횟수 한도** 를 정할 수 있습니다.
"5번까지만 시도하고, 그래도 실패하면 별도의 우체통에 넣어둬." 이 별도 우체통이 **DLQ (Dead Letter Queue)** 입니다.

```
[정상 흐름]
편지 ──→ order-changed 우체통 ──→ 쿼리 서비스 (성공) ✓

[실패 흐름]
편지 ──→ order-changed 우체통 ──→ 쿼리 서비스 (실패 1번)
                                  ↓ 다시 시도
                              ──→ 쿼리 서비스 (실패 2번)
                                  ↓ ...
                              ──→ 쿼리 서비스 (실패 5번)
                                  ↓
                            order-changed-dlq 로 이동 → 사람이 확인
```

DLQ 에 메시지가 쌓이면 **알람** 이 울려서 개발자가 살펴봅니다.
원인을 고치고 다시 정상 우체통으로 돌려보내거나, 무시할 메시지면 그냥 지웁니다.

### 코드에서는?

대부분 **AWS 인프라 설정** 의 영역입니다. 우리 프로젝트에서는 `infra/sqs.tf` 파일에 정의되어 있습니다.

```hcl
resource "aws_sqs_queue" "main" {
  name = "${each.value}-changed"
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.dlq[each.value].arn
    maxReceiveCount = 5      # ← 5번 실패하면 DLQ 행
  })
}
```

이 Terraform 파일 한 번 적용하면 4개 도메인 각각에 본 우체통 + DLQ + 알람이 다 만들어집니다.

자바 코드 쪽에서는 `SqsErrorHandlerConfig.java` 에서 **실패 시 깔끔하게 로그를 남기기만** 하면 됩니다.
재시도 정책은 AWS 가 알아서 합니다.

## 2. 두 번 온 편지 — 멱등성(idempotency)

### 문제

SQS 는 **"같은 메시지를 적어도 한 번 이상 보내준다"** 를 보장합니다.
바꿔 말하면 **두 번 보낼 수도 있다** 는 뜻입니다. 네트워크가 잠시 끊기면 일어나는 흔한 일.

같은 회원 변경 편지가 두 번 처리되면? 우리 코드에선 결과가 같아서 문제 없을 수 있지만,
만약 "처리하면서 슬랙으로 알림 보내기" 같은 부작용이 있다면 알림이 2번 갑니다.
원치 않는 상황입니다.

### 해결: 처리한 편지 번호를 적어두기

매 편지마다 **고유한 번호(eventId)** 가 있습니다.
편지 처리하기 전에 **"이 번호 이미 처리했나?"** 부터 확인합니다.

이를 위해 `processed_event` 라는 작은 테이블이 있습니다.

```
processed_event 테이블
┌──────────────┬──────────────┬──────────────────┐
│ event_id     │ consumer     │ processed_at     │
├──────────────┼──────────────┼──────────────────┤
│ abc123       │ orderList    │ 10:00:00.500     │
│ abc124       │ orderList    │ 10:00:01.200     │
└──────────────┴──────────────┴──────────────────┘
```

처리할 때마다 한 줄을 추가합니다.
이미 같은 (event_id, consumer) 쌍이 있으면 추가 실패 → "이미 처리한 거구나" 알고 그냥 건너뜁니다.

### 코드 한 줄

`IdempotencyService.tryClaim()` 이 그 역할입니다.

```java
if (!idempotency.tryClaim(event.eventId(), CONSUMER)) {
    return;   // 이미 처리됨, 끝.
}
// 처음 본 편지라면 여기 이후 코드가 실행됨
```

`tryClaim` 안에서는 `INSERT IGNORE` SQL 을 씁니다.
- 처음 본 eventId 면 → 한 줄 추가 + true 반환 → 계속 진행
- 이미 있는 eventId 면 → 추가 실패 + false 반환 → 건너뜀

### 왜 (event_id, consumer) 쌍인가?

여러 화면이 같은 도메인 편지를 구독할 수 있기 때문입니다.

- "주문 목록 화면" 도 회원 변경 편지를 받음
- "회원 마이페이지 화면" 도 회원 변경 편지를 받음

같은 편지여도 **각 화면별로 따로** 처리됐는지 추적해야 합니다.
그래서 consumer 컬럼에 화면 이름을 넣습니다.

### 자동 청소

`processed_event` 테이블이 무한정 커지면 안 됩니다.
`ProcessedEventCleanupJob` 이 매일 새벽 4시에 **7일 지난 기록을 지웁니다.**
같은 편지가 7일 뒤에 또 올 일은 거의 없으니 충분합니다.

## 3. 거꾸로 온 편지 — 섹션별 timestamp

### 문제

회원 등급이 빠르게 두 번 바뀌었습니다.

- 09:00:00 — "실버 → 골드" 편지 발송
- 09:00:01 — "골드 → 플래티넘" 편지 발송

근데 네트워크 사정으로 두 번째 편지가 먼저 도착했고, 첫 번째 편지가 나중에 도착했습니다.

만약 순서 무시하고 처리하면? "플래티넘" 으로 바뀐 다음 "골드" 로 되돌려져요. 잘못된 결과입니다.

### 해결: "이 영역 최신 도장 시각보다 새 편지일 때만 적용"

각 도메인 영역에 `xxxSectionUpdatedAt` 도장이 있다고 했죠.
새 편지가 오면 도장 시각 비교부터 합니다.

```sql
WHERE member_id = 42
  AND (member_section_updated_at IS NULL
       OR member_section_updated_at < :occurredAt)
```

- 도장이 비어있으면 (= 처음 받는 편지) → 적용
- 도장 시각이 새 편지 시각보다 이전이면 → 적용
- 도장 시각이 새 편지 시각보다 나중이면 → 안 적용 (이 줄 0개 업데이트되고 끝)

이 패턴을 **last-write-wins (최신 글쓰기가 이김)** 이라고 부릅니다.
**SQS 가 순서를 보장 안 해도 결과는 항상 최신 상태로 수렴** 합니다.

위 예제를 다시 보면:

```
09:00:01 편지가 먼저 도착 → 도장 비어있음 → 적용 → 도장=09:00:01
09:00:00 편지가 나중 도착 → 도장(09:00:01) > 편지(09:00:00) → 안 적용 ✓
```

순서가 거꾸로 와도 최종 상태는 "플래티넘" 으로 정확하게 유지됩니다.

---

## 한 줄 요약

- **DLQ:** 5번 실패한 편지는 따로 보관. 사람이 본다.
- **멱등성:** 편지 번호 적어두고, 같은 번호 두 번째부터는 무시.
- **순서 보장:** 영역마다 도장 찍어두고, 도장보다 새 편지일 때만 적용.
